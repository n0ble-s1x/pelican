# Code Audit · 2026-05-03

Snapshot review of the upload pipeline, MTP backend, history, and tests.
What's solid, what's worth fixing, what we deliberately skipped.

## Fixed in this pass

### 1. `--no-transcode` filename safety
**Was:** `transfer::sanitize_name` only stripped FAT-hostile characters
(`/\:*?"<>|`). It did not enforce the 56-character cap that the transcode
path applies via `transcode::sanitize_filename_stem`. Users uploading
already-supported formats (`.mp3` / `.m4a` / `.aac` / `.wav`) with long
filenames hit Garmin's silent-rejection / broken-stub failure mode despite
the format being natively supported.

**Now:** `transfer::sanitize_name` extracts stem + extension, runs the stem
through `transcode::sanitize_filename_stem` (now `pub`), then re-attaches
the lowercased extension. Same sanitizer covers both code paths.

### 2. Upload memory peak
**Was:** `mtp::MtpRsBackend::upload` called `std::fs::read(local)` to load
the entire file into memory, then `bytes::copy_from_slice` to build a Vec
of `Bytes` chunks for streaming. Peak memory ≈ 2× file size before the
first byte left the wire. A 100 MB audiobook held 200 MB.

**Now:** `futures::stream::poll_fn` reads `CHUNK` (256 KB) at a time
directly from `std::fs::File`. Peak memory ≈ CHUNK + mtp-rs internal
buffers. Verified end-to-end on hardware.

### 3. Test coverage gap
**Was:** Only one unit test (`playlist::round_trip`) plus four integration
tests covering invariants. No tests for any of the sanitizers despite them
encoding hard-won protocol knowledge.

**Now:** 25 unit tests across `playlist.rs`, `transcode.rs`, `transfer.rs`
covering: filename stem sanitization (cap, char allowlist, dash collapse,
outer trim, never-empty), tag value sanitization (©®™℗ + controls, accent
preservation), audio classification, supported-format classification,
directory-tree expansion (flatten + nested), playlist parse edge cases
(CRLF, blanks, comment variants, empty), playlist serialize for both
`PathStyle` variants.

## Worth doing later

### 4. Filename collision detection
When two source files sanitize to the same target name, the watch
**corrupts both files** rather than cleanly overwriting (verified
2026-05-03: uploading `11 - Iva Davies, Christopher Gordon, Richard
Tognetti - Ghost of Time - Tognetti Into the Fog.mp3` over an existing
file with the same 56-char-truncated stem produced **2 broken stubs and
zero readable files**). The existing file becomes unreadable too — the
collision is destructive on the watch side.

**Fix:** Before upload, list the target directory and check for an
existing file with the same sanitized name. If present, append a numeric
suffix (`-2`, `-3`) until unique. Alternative: hash-based suffix using a
prefix of the source-file SHA-256. Suffix needs to fit inside the 56-char
cap, so reserve ≤4 chars by truncating the stem to 52 first.

### 5. List-directory parallelism
`MtpRsBackend::list_dir` issues `session.get_object_info(handle).await`
sequentially for every handle. On a watch with hundreds of music files
this is the slowest operation in the GUI. Parallelizing with
`futures::stream::iter(handles).buffered(N)` would help, BUT mtp-rs's
session is a single USB endpoint and our experience is that concurrent
requests against the Garmin responder wedge it. Verify against hardware
before changing.

### 6. Upload retries
Garmin firmware is known-flaky on `SendObjectInfo` (`better-sync` retries
3× with 1s sleep between). Pelican has no retry — a single transient
failure becomes a job failure. Adding a single 1s-spaced retry would
likely catch the cases without papering over real wedge states. Risk: a
retry-storm on a bad watch state can extend wedge time. Worth adding
behind a flag (`--retry`) so users opt in.

### 7. CLI session reuse
`cli::run_headless` opens a fresh MTP session for each phase
(`--delete`, `--list-playlists`, `--create-playlist`). Each open costs
~2-5s (USB enumerate, OpenSession, GetStorages, GetStorageInfo). For
single-flag invocations this is invisible; for combined flags it
compounds. Trade-off: holding one session across phases means a wedge in
phase 1 corrupts phase 2. Per-phase open is more defensible; defer.

### 8. `history.rs` write amplification
`history::record` rewrites the whole JSON file on every upload. With
1000-record cap and average track size, that's a few KB written ~50
times per multi-album batch. Negligible on SSD; would matter on
spinning rust or a battery-conscious laptop. Append-only log + periodic
compaction is the textbook fix; not currently warranted.

### 9. `transcode::cache_dir` and `history::data_dir` duplication
Both functions recompute the same XDG_*_HOME / HOME fallback ladder.
Could be factored into a shared `paths` module. Cosmetic.

### 10. `MtpRsBackend::write_raw` chunk pre-allocation
Same all-in-RAM issue as `upload` had — caller passes `bytes: &[u8]`
slice and we build a chunked Vec. For playlist files this is small (~few
hundred bytes), so the cost is trivial. Not worth fixing until something
calls `write_raw` with large buffers.

## Deliberately not changed

- **Per-file MTP session in `transfer::run_jobs_per_file`** — looks
  wasteful but is **load-bearing**: Garmin firmware silently rejects most
  uploads after the first 1-2 in a session. Verified, documented, leave
  as-is.
- **`set_split_header_data(true)` always-on** — required by Garmin's
  microsoft.com-vendor MTP responder. Documented in
  `docs/garmin-mtp.md`.
- **Flatten-by-default in `expand_inputs_with`** — Garmin firmware bug
  on listing freshly-created subfolders. Documented.
- **`PathStyle::UppercaseWithPrefix` left in source despite being unused
  in production code paths** — kept as a documented seam for the
  playlist work. `#[allow(dead_code)]` on the variant.
